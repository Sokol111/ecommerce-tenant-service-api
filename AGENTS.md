# AGENTS.md

> This repo is one member of the `ecommerce-*` multi-repo workspace. See the workspace-root
> `CLAUDE.md` for the release-then-bump rule, `go.work` resolution, and dev-container notes.

## What this repo is

The **API contract** for `ecommerce-tenant-service`. Tenant-service owns multi-tenancy for the
whole platform, so this module is imported by **every other service** — its gRPC/Connect client,
its Kafka event types, and two ready-to-wire `fx` helper modules all live here.

Unlike a pure proto repo, this one ships **hand-written Go under `pkg/`** on top of the generated
code. That Go is the main thing to edit here; the generated `gen/` code is not.

## The two source layers

1. **`proto/` → `gen/` (generated, never hand-edit).** `gen/go` (the published Go module) and
   `gen/typescript` (`@sokol111/ecommerce-tenant-service-api`, consumed by the Nuxt UIs) are both
   fully generated by `buf`. A repo-root hook also blocks edits under `gen/`. Change `.proto` and
   run `make generate`.
   - `proto/tenant/v1/` — RPC services: `TenantService` (CRUD + `GetEnabledTenantSlugs`) and
     `TenantRegistrationService` (self-service signup saga, tracked via `RegistrationStatus`).
   - `proto/tenant/events/v1/` — Kafka event schemas: `TenantUpdatedEvent` (create/update) and
     `TenantDeletedEvent`. Field-level validation is declared inline with `buf.validate`.

2. **`pkg/` (hand-written, the real work).** Small `fx`-wired helpers so consuming services get
   tenant integration for free:
   - `pkg/client` — `Module()` provides a native gRPC `TenantServiceClient`, config from koanf key
     `tenant.grpc`.
   - `pkg/provider` — adapts that client to `commons/pkg/tenant.SlugsProvider` (used by services to
     discover which tenant DBs exist).
   - `pkg/consumer` — Kafka consumer `Module()` that reacts to tenant events: runs a DB migration
     on `TenantUpdatedEvent` (when enabled) and deferred cleanup on `TenantDeletedEvent`, via
     `commons/pkg/tenant.Lifecycle`. Services add their own cleanup by registering
     `tenant.Cleaner` implementations into the `tenant_cleaners` fx group (see the doc comment on
     `Module()`).
   - `pkg/events` — `TopicFor(msg)` maps event proto types to their Kafka topic
     (`tenant.tenant.events`).

## Commands

```bash
make generate          # buf lint + generate Go (Connect+gRPC) + TypeScript + event stubs — run after ANY .proto edit
make lint              # buf lint only
make format            # buf format -w
make connect-breaking  # detect breaking proto changes vs main branch
make tidy              # go mod tidy
make connect-install-tools   # install buf + protoc-gen-{go,connect-go,go-grpc} at pinned versions
```

There is **no `go test` / `go build` target** — this module has no runnable binary and (currently)
no tests; it is a library consumed by other repos. Validate changes with `make generate` (which
lints proto and regenerates), then build a consuming service against it via the workspace `go.work`.

## When editing

- **Adding/changing an RPC or event field:** edit the `.proto`, then `make generate`. If you touch
  a Kafka event message, also update `topicMap` in `pkg/events/topics.go` — it is keyed by proto
  full-name and **panics** on an unregistered message, so a new event type without a topic entry
  fails at runtime, not compile time.
- **Breaking proto changes** are gated: releases run a breaking-change check (skippable only via the
  `workflow_dispatch` `skip_breaking` input). Prefer additive changes.
- **Releasing:** bump the `VERSION` file and push to `master` — that path triggers
  `.github/workflows/release.yml`, which tags and publishes both the Go module and the TS package.
  Consuming services then bump their `go.mod` (release-then-bump; local dev sees changes immediately
  through `go.work`).
